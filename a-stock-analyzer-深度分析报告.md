# A 股股票深度分析 Skill — 深度分析与优化建议

## 一、整体评价

这是一个结构完整、可用性较高的 A 股分析 Skill。它涵盖了基本面、技术面、资讯面三大分析维度，配备了自动化的数据拉取、评分计算和 HTML 报告生成功能。整体架构清晰，代码可读性好。

综合评分：**70/100** — 框架扎实，但在健壮性、分析深度和用户体验上有较大提升空间。

---

## 二、架构分析

### 文件结构

```
a-stock-analyzer/

├── scripts/
```

**优点：** 遵循了 Skill 的三层架构（SKILL.md → scripts → references），职责分离清晰。

**问题：** `stock_analyzer.py` 达到 1190 行，过于庞大。三个分析器类（`FundamentalAnalyzer`、`TechnicalAnalyzer`、`SentimentAnalyzer`）应拆分为独立模块。

## 三、SKILL.md 分析

### 3.1 Description（触发描述）

当前描述覆盖面广，列举了 7 种触发场景，且有"即使用户只是随口问一句也应该触发"的兜底语句，这一点做得好。

**可改进：**
- 缺少负面触发边界（什么时候不该触发），比如港股、美股、基金等场景
- 缺少对比 `data:analyze` 等其他 Skill 的边界说明
- 建议增加更多口语化触发词，如"帮我看看盘"、"这个票能不能入"、"XX 值不值得拿"

### 3.2 工作流程

四步工作流清晰（确认目标 → 安装运行 → 生成报告 → 解读总结），第四步要求 Claude "像一个资深分析师那样给出有洞察力的点评" 是很好的指引。

**可改进：**
- 缺少对 Token 的缓存/复用机制说明（每次都要用户提供 Token 很烦）
- 没有指导 Claude 在脚本报错时如何降级处理
- `<skill-path>` 占位符可能让 Claude 困惑，建议使用更明确的路径引用方式

## 四、stock_analyzer.py 代码分析

### 4.1 优点

1. **重试机制**：`safe_api_call` 实现了指数退避重试，能应对 Tushare 频率限制
2. **评分体系设计合理**：`score_range` 函数配合阈值打分，逻辑清晰
3. **代码注释充分**：每个分析子模块都有 print 输出，方便调试
4. **股票代码自动补全**：能识别 SH/SZ/BJ 后缀

### 4.2 问题与缺陷

#### 🔴 严重问题

**1. 无行业对比基准**

运营效率分析中直接给了硬编码阈值（如 `ar_turn > 6`），但不同行业差异极大：银行的应收周转率和制造业完全不同。偿债能力同理——银行的资产负债率天然在 90% 以上，当前逻辑会给出极低分。

```python
# 当前代码（第216-223行）
score = 50  # 运营效率需要行业对比，这里给中性基准
if ar_turn is not None and ar_turn > 6:
    score += 10
```

**建议：** 引入行业分类（Tushare 的 `stock_basic` 已提供 `industry` 字段），针对不同行业调整阈值，或至少在评论中注明"该指标需结合行业特性判断"。

**2. 估值分析只看了 3 年历史分位**

PE/PB 的历史分位数只取了 3 年数据。对于周期股，3 年可能刚好在周期顶部或底部，得出的"历史低估"结论可能是误导。

```python
start_date=(datetime.now() - timedelta(days=365*3)).strftime('%Y%m%d')
```

**建议：** 默认使用 5 年数据，对上市不足 5 年的股票降级到 3 年并注明。

**3. 成长性分析过于简化**

只取了最新一期的同比增速，没有计算 CAGR（代码中预留了变量 `rev_yoy_list` 但未使用）。

```python
# 第169行：计算了 rev_yoy_list 但完全没用到
rev_yoy_list = df['or_yoy'].head(12).dropna().tolist() if 'or_yoy' in df.columns else []
```

**建议：** 实现 3 年营收/净利润 CAGR 计算，并纳入评分。

#### 🟡 中等问题

**4. 背离检测算法过于粗糙**

```python
price_new_high = recent['close'].iloc[-1] >= recent['close'].iloc[-30:].max() * 0.98
dif_no_new_high = recent['DIF'].iloc[-1] < recent['DIF'].iloc[-30:].max() * 0.9
```

用固定的 2%/10% 容差判断"新高/新低"不够精确。实际 MACD 背离判断需要找到明确的波峰/波谷对来比较，而非简单比较最近值和区间极值。这会产生大量假信号。

**5. 数据缺失时默认 50 分**

所有子模块在数据缺失时返回 50 分（中性分），但 50 分会被纳入加权平均，可能拉低或拉高真实分数。

**建议：** 数据缺失时不应参与加权计算（当前 `_calculate_score` 已有 `total_w` 逻辑，但子模块内部仍返回 50 分的中性评分参与了计算）。

**6. 缺少异常值处理**

```python
latest = df.iloc[0]
roe = latest.get('roe_waa')
```

Tushare 返回的数据可能包含极端异常值（如 ROE > 200% 或为负数的情况），代码没有做任何裁剪或异常标记。

**7. 资讯面缺少「股东增减持」维度**

SKILL.md 中提到了 `stk_holdertrade`（股东增减持），但 `SentimentAnalyzer` 中完全没有实现。这是一个重要的信号维度。

#### 🟢 小问题

**8. 硬编码的 sleep 时间**

```python
time.sleep(0.5)  # 第1091-1093行
```

在三个分析器之间固定等待 0.5 秒。Tushare 的频率限制取决于用户积分等级，应让 `safe_api_call` 的重试机制自行处理。

**9. moneyflow 数据单位混淆**

```python
'当日主力净流入(万)': round(latest_net / 10000, 2)
```

Tushare 的 `moneyflow` 接口返回的 `net_mf_amount` 单位是"万元"，这里又除以了 10000，变成了"亿元"，但标签写的是"万"。

## 五、generate_report.py 分析

### 5.1 优点

- HTML 报告设计美观，使用了现代 CSS（grid、圆角、渐变）
- 响应式布局（media query 适配移动端）
- 使用 Chart.js CDN 绘制图表，无需额外安装

### 5.2 问题

1. **CDN 依赖可能失败**：使用了 `cdnjs.cloudflare.com` 的 Chart.js 4.4.1，在网络受限环境下报告会无法渲染图表。建议提供降级方案（纯文本表格）。

2. **RSI 图表的参考线依赖 annotation 插件**：代码中使用了 `annotation` 插件画 RSI 的 30/70 参考线，但没有引入 `chartjs-plugin-annotation` CDN，这些线在实际渲染中不会显示。

3. **缺少雷达图**：代码中计算了 `radar_data`（第215-219行）但从未使用。三维评分的雷达图是一个很直观的可视化，应该补上。

4. **没有 K 线图**：技术分析报告中只有折线图，缺少真正的 K 线（蜡烛图），对于技术分析来说缺少专业感。

## 八、总结

这个 Skill 的核心骨架已经很好——三维分析框架、自动化评分体系、HTML 可视化报告，这些都是对的方向。主要的改进空间在于：

1. **分析准确性**：行业差异化是最大的短板，一套通用阈值打遍所有行业必然不准
2. **数据完整性**：SKILL.md 承诺的功能（CAGR、股东增减持）代码中未完全实现
3. **细节 bug**：moneyflow 单位错误、RSI annotation 插件缺失、雷达图数据未使用等

建议按优先级分阶段迭代，先修复 bug 和数据错误，再增强分析深度，最后优化体验。
